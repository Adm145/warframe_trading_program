[X] 5. Order mutations - POST /v2/order, PATCH /v2/order/{order_id},
    DELETE /v2/order/{order_id}  (CONFIRMED LIVE - full create/update/delete
    cycle tested end-to-end against a real, throwaway order)
    NOTE: path is /v2/order, SINGULAR - inconsistent with the read
    endpoint /v2/orders/my (plural). Easy to get wrong.
    WIRED INTO EXPRESS: Backend/routes/order.route.js +
    Backend/controllers/order.controller.js - postNewOrder (POST
    /order/add/:item_name), updateOrderById (PATCH /order/update/:order_id),
    deleteOrderById (DELETE /order/delete/:order_id), and closeOrderById
    (POST /order/close/:order_id, reduces quantity or deletes outright if
    no quantity given / quantity would hit zero).

    a) POST https://api.warframe.market/v2/order
       Auth: same "Authorization: Bearer <token>" as /v2/me, /v2/orders/my
       Required body fields (revealed via the validation error from an
       empty {} body): itemId, type ("sell"/"buy"), platinum, quantity,
       and rank - rank is only required for items with ranks (mods,
       rivens, etc; app.field.required if omitted on a rankable item;
       omit entirely for non-rankable items).
       Live test: POST itemId=54a74454e779892d5e5155cc (blind_rage),
       type=sell, platinum=600, quantity=1, rank=10
       -> 200 {"id":"6a4e46e12202c9dc0adf1d05","type":"sell","platinum":600,
               "quantity":1,"perTrade":1,"rank":10,"visible":false,
               "createdAt":...,"updatedAt":...,"itemId":"54a74454e779892d5e5155cc"}
       New orders default to visible:false (hidden) - matches the
       reference bot's pattern of posting hidden-by-default.

    b) PATCH https://api.warframe.market/v2/order/{order_id}
       IMPORTANT: this is PATCH, not PUT. PUT returned 405 Method Not
       Allowed on both /v2/order/{id} and /v2/orders/{id}; only
       PATCH /v2/order/{id} (singular) worked.
       Live test: PATCH platinum=500, quantity=2 on the order above
       -> 200, platinum/quantity updated, rest unchanged, updatedAt bumped.

    c) DELETE https://api.warframe.market/v2/order/{order_id}
       Live test: DELETE on the same order -> 200, echoes the deleted
       order back in the response. Verified gone via a follow-up
       GET /v2/orders/my (back to only the original 2 real orders).

    Reference: postOrder, updateListing, deleteOrder in
    AccessingWFMarket.py - the actual trading actions called by
    compareLiveOrdersWhenBuying / compareLiveOrdersWhenSelling.

[X] 6. Auth/login - POST /auth/unlock (bcrypt-gated, .env-sourced
    credentials - FINAL DESIGN, supersedes the original plan below)

    Real market credentials never touch the frontend at all, not even
    once. Final architecture:
      - .env holds WFM_EMAIL, WFM_PASSWORD (real market credentials,
        backend-only) and ACCESS_KEY_HASH (a bcrypt hash of a separate,
        unrelated personal access key chosen by the user - not the
        market password).
      - POST /auth/unlock takes { accessKey } from the frontend. Backend
        compares it against ACCESS_KEY_HASH via bcrypt.compare
        (Backend/bcrypt/accessKey.util.js -> compareAccessKey). Only on
        a match does it run the v1 sign-in internally using
        WFM_EMAIL/WFM_PASSWORD from .env, save the resulting token via
        session.util.js's saveSession, and respond with
        { ingameName, timestamp } only - the token itself never leaves
        the backend.
      - Backend/bcrypt/generateHash.js is a one-off CLI script
        (node bcrypt/generateHash.js "your-key") to generate the
        ACCESS_KEY_HASH value once, run manually, not imported by the app.
      - The original signInController (raw email/password read from the
        request body) was implemented first, then removed entirely once
        unlockController fully replaced it - keeping both would have left
        signInController as a completely unauthenticated back door
        achieving the same result with no access-key gate at all.
      - Frontend implication: only ever prompt for the access key, never
        for real market credentials. Show the prompt when any
        authenticated call returns 401 (token expires after ~60 days per
        the JWT's own exp/iat claims), call POST /auth/unlock, then
        retry/refresh.

    LIVE-TESTED: wrong access key -> 401 "Invalid access key."; correct
    access key -> 200 { ingameName, timestamp }, confirmed session.json
    updated with a fresh real token.

    The (a)/(b)/(e) notes below are the original research into how v1
    auth works and are still technically accurate and still used inside
    unlockController today (same request shape, same "JWT " -> "Bearer "
    conversion). (c)/(d) described the reference repo's v1-only pattern
    and don't reflect the final code path - kept for context only.

 [X]  a) POST https://api.warframe.market/v1/auth/signin
        Headers: Content-Type: application/json; utf-8,
                    Accept: application/json,
                    Authorization: JWT   (literal, no token yet),
                    platform: pc/ps4/xbox/switch (lowercase),
                    language: en
        Body: { "email": "<email>", "password": "<password>",
                "auth_type": "header" }
        "auth_type": "header" tells warframe.market to return the token in
        a response header instead of an HttpOnly cookie, since a script
        can't read cookies the way a browser session would.
        Still exactly what unlockController sends, just with email/
        password sourced from process.env instead of req.body.

 [X]   b) Token comes back in the response, not a cookie:
        response.json().payload.user.ingame_name -> in-game name
        response.headers.Authorization  -> "JWT eyJhbGciOi..."
        Still used as-is in unlockController.

 [X]   c) (historical only) Every authenticated v1 request reusing a
        fixed v1 header block (auth_type/platform/language/Authorization:
        JWT ...) - this pattern from the reference repo's WarframeApi
        class is not used anywhere in the final code; every authenticated
        call in this project targets v2 with a Bearer token instead (see e).

 [X]    d) (historical only) The reference repo has no refresh logic - one
            token per process lifetime, no re-auth on 401. This project's
            equivalent is the frontend catching a 401 and re-prompting for
            the access key (see "Frontend implication" above) rather than
            an automatic silent refresh.

 [X]    e) CONFIRMED LIVE: v2 endpoints do NOT accept the v1
            "Authorization: JWT <token>" header format. Tested directly:
              Authorization: JWT <token>        -> 401 app.errors.unauthorized
              Authorization: Bearer <token>      -> 401 app.errors.unauthorized
              Authorization: Bearer <raw token, "JWT " prefix stripped>
                                                  -> 200, real data
            So the v1 sign-in response's token string ("JWT eyJ...") must
            have the "JWT " prefix stripped before being reused on v2
            calls, and must be sent as "Bearer <rest of string>" instead
            of "JWT <rest of string>". This conversion is not documented
            anywhere in the reference repo (which never talks to v2) or in
            the community OpenAPI spec - found only by live-testing, and
            is exactly what unlockController does today
            (response.headers.get("authorization").replace("JWT", "Bearer")).
